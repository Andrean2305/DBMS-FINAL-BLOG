This is our last table, which is the transaction table. The usage of this table is to store all the user transaction history. The history itself can be used by the user and also the admin. However, for the user they only can see their own history. While the admin can see all the transactions to see and compare to improve our recommendation system (our main function in the application). For the date we use a simple code to find the local time of the user, so they can have their own actual date. We try to gather all the transaction information to make personalized recommendations, and not just based on some information such as car price or gas emission. In the future, we want to make our users get a recommendation based on their liking.

  • GUI/APPLICATION

This is the first interface that the user(customer/admin) will see when opening the application. They can input their username and password and click login if they already have the username. Or they can click sign up to make a new username and password.

They can input username and password using symbols, letters, or numbers. However they cannot input a username that people already took. For example, my username is a. Other people can’t create a username with a letter a, because I already took the username. An admin can put the unique admin code to make an admin id. Finally, they can also choose what car they like, they either choose a practical car or a performance car. This car preference will be used for our user recommendations.

This is the interface when the user already logins to the application. There are 3 features in the main interface, they are profile button, shop button, and recommendation button.

As you can see, in the recommendation interface, people will be recommended based on what car preference they chose.

This is our shop interface. They can choose the brand name and after that they can choose the model based on the brand. We connect the brand table and car table to make this. After the user chooses, they can click the exclamation symbol to find the fuel information. We make the information using table joins to take information from the car table and fuel table. After they choose the car, and the price and other information match with their preferences, they can click the buy button and then finish.

Now, in the profile interface, users can change their password, and their car preference. However, they cannot change the username, because username is the unique key. Actually, we can allow users to change the username and change user id to be primary, however, based on our research. Most of the applications do not allow people to change usernames. And if they already input and want to change, they can click the save button. And then, there are also history buttons.

In the history, users can check their transaction history, they can check the date, what car they bought, and the price.

This is our admin interface, they can check dealer info, user database, and they can modify car information.

In the car modifier interface, admin can add new car models based on the brand. The main features of this interface are adding and deleting(kinda like undoing because you can’t choose yet). And then, changing and adding stock. as you can see the 2 buttons at the top.

After they click it, the mini interface will pop out. And then, they need to choose what car model they want to change, after that they can input the car id and input how much they want to add to the stock. They can also change the stock. It is an important feature because the stock is connected to the sales system, which mean after user click buy in the shop interface, the system will check the stock first, and if there are stocks for the model the user want to buy, they decrease the stock and add to total sales in dealer info in the brand table.

Admin also can see the customer id of the username and what car preference they chose. And at the right bottom of the interface, there is a transaction button.

While customers only can see their own transaction, an admin can see all of the transaction history from all of the customers.

Lastly, this is the dealer info interface. The job of this interface is to check the dealer information. So the admin can see dealer information if they want to contact the dealer. They also can check total sales that can be used as an information for whether they should stock the car brand more or less. And then, same as the car modifier, this interface also allows the admin to add dealer information to the database.

  • CONCLUSION

    • With the recommendation as the main feature of our application, customers or people who want to buy a car can easily pick what car they want to buy with all of the information from our application.
